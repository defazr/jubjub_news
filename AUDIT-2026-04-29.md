## 7-2. 결론 요약

### 1. `summarize=true`가 timeout 원인일 가능성 — 사실상 확정

**코드 근거** (`src/app/api/news-ingest/route.ts:499-542`):

- `?summarize=true`일 때 `prepared` 배열의 **모든 신규 기사**에 대해 `Promise.allSettled`로 Anthropic API (Claude Haiku) 동시 호출
- prepared 크기는 RapidAPI에서 fetch한 9개 query (trending + 8 카테고리) 결과에서 중복/스팸 필터링 후 남은 양
- CLAUDE.md 운영 로그: `fetched: 217`, `inserted: 31` → AI 호출 약 30~50건/run
- 추가로 RapidAPI 9회 fetch (병렬) + Supabase dedup query + Supabase upsert 50개씩 batch

**타이밍 추정**:

- AI 30~50회 병렬: 보통 8~20초, 큐잉/지연 시 30초+
- RapidAPI fetch 병렬: 3~10초 (외부 API 변동)
- Supabase: 1~3초
- **합계가 Netlify Pro 26초를 자연스럽게 초과** → 41초 502 패턴과 일치

**중요**: 코드의 `maxDuration = 60`은 Vercel용 설정. Netlify는 Pro 26s / Free 10s가 강제. 코드의 maxDuration은 Netlify에서 무시됨.

### 2. backfill-summaries가 이미 존재하는지 — 예 (이미 존재)

`news-ingest.yml` line 25-32에 이미 step 2로 정의되어 있음:

```
/api/backfill-summaries?secret=...&limit=50  --max-time 180
```

→ **신규 API 엔드포인트 만들 필요 없음**. 라우트도 `src/app/api/backfill-summaries/route.ts`에 구현 완료.

### 3. 같은 workflow 순차 호출 vs 별도 workflow — 별도 workflow 분리가 안전

**현재 (같은 workflow 순차)의 문제**:

- step 1 (news-ingest with summarize=true)이 `exit 1`로 실패하면 step 2 (backfill-summaries)는 **실행 자체가 안 됨**
- 즉 ingest가 502로 죽는 동안 backfill도 같이 죽음 → AI 요약이 **이중으로 막힘**
- step 1이 26초 가까이 점유 후 502 → step 2 시작 시점부터 다시 26초 카운트 시작이지만, step 2도 50건 처리 시도하면 20~40초 → 또 502

**별도 workflow의 장점**:

- ingest는 가볍게 (summarize=false) 빠르게 끝나도록 분리 (예상 5~10초)
- backfill-summaries는 별도 cron으로 작은 limit (예: 10~15) × 더 자주 (15분마다) 호출
- 한쪽 실패가 다른 쪽 차단하지 않음
- Free 10s 전환 시에도 limit 조정만으로 대응 가능

### 6. 41초 502 실패 — workflow / step 매핑

**실제 GitHub Actions 로그 접근 불가** (도구 제약). 따라서 **추론 기반 매핑**이며, 정확한 확인은 로그 직접 조회가 필요:

**추론 결과**: `news-ingest.yml` → **step 1 "Trigger news ingest"** (`summarize=true` 호출)에서 발생.

**근거**:

- step 1만 `summarize=true`를 호출 → 26s 자연 초과 케이스의 유일한 후보
- step 1이 `[ "$response" -ne 200 ] && exit 1`로 fail-fast → 워크플로 전체가 step 1 종료 시점에 실패
- 41s = Netlify Pro 26s timeout + edge layer 지연/응답 버퍼링 (GPT 분석과 일치)
- 패턴이 정확히 41초로 반복 → 플랫폼 timeout 성격 (random 네트워크 오류 아님)

**검증 방법** (사용자가 직접 확인):

- GitHub Actions → "News Ingest Cron" → 4/28 21:03 UTC, 4/29 02:34 UTC 실패 run 클릭
- "Trigger news ingest" step에서 빨간 X면 추론 확정
- "Trigger backfill summaries" step이 회색(skipped)이면 step 1에서 죽은 것 확정
